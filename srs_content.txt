1. Introduction
1.1 Purpose
This Software Requirements Specification (SRS) defines the complete functional and non-functional requirements for DevPilot, an AI-powered terminal-based coding agent. The document serves as the primary reference for design, development, testing, and project planning activities following the Agile Scrum methodology. It is intended for the developer, academic reviewers, and any stakeholders evaluating the system.
1.2 Scope
DevPilot is a command-line agent that accepts natural-language coding tasks and fulfils them autonomously through an agentic loop that reads files, writes code, executes shell commands, and delegates to specialist agents. The system integrates two open protocols:
Model Context Protocol (MCP) ΓÇö for dynamic, pluggable tool discovery from external MCP servers.
Agent-to-Agent Protocol (A2A) ΓÇö for peer-agent communication enabling multi-agent workflows.
DevPilot is model-agnostic: it supports Anthropic Claude, OpenAI GPT models, and any OpenAI-compatible local model (Ollama). It is designed as a portfolio-grade open-source project demonstrating advanced agentic AI system design.

2. Overall Description
2.1 Product Perspective
DevPilot is a standalone CLI application inspired by Claude Code (Anthropic) and Aider. It extends the category by adding two open-protocol integrations not present in existing tools:
MCP Client ΓÇö connects to any conforming MCP server at startup and exposes its tools to the model without code changes.
A2A Dual-Role ΓÇö acts simultaneously as an orchestrator (delegates to peer agents) and as a worker (accepts tasks from external agents via HTTP).
The system is model-agnostic by design. The provider abstraction layer normalises API differences so the agentic loop, tool system, and A2A layer are independent of which model is in use.
2.2 Product Functions Summary
Natural-language coding task execution via agentic loop.
File read/write, bash execution, and code search tools.
Dynamic tool discovery and execution via MCP protocol.
Multi-agent task delegation and reception via A2A protocol.
Support for Anthropic, OpenAI, and local (Ollama) model providers.
Persistent session history and CI-friendly one-shot mode.
Permission guard for all destructive operations.

2.4 Operating Environment
Operating Systems: macOS 13+, Ubuntu 22.04+, Windows 11 with WSL2.
Runtime: Python 3.11 or later.
Network: Internet access for model API and remote MCP servers; local mode supported via Ollama.
Terminal: Any ANSI-colour-capable terminal (iTerm2, Windows Terminal, GNOME Terminal, etc.).
2.5 Design and Implementation Constraints
The agentic loop must be implemented without threading ΓÇö all async I/O via Python asyncio.
No third-party agent framework (LangChain, LlamaIndex) may be used in the core loop; protocol integrations are direct SDK calls.
All API keys must be sourced from environment variables; hardcoded credentials are forbidden.
The A2A server must comply with the Agent-to-Agent Protocol v0.2 specification.
MCP client must comply with the Model Context Protocol specification using the official Python MCP SDK.
2.6 Assumptions and Dependencies
The model provider exposes a tool-use / function-calling API compatible with Anthropic or OpenAI schemas.
MCP servers are reachable via stdio transport or HTTP SSE transport at startup.
Peer A2A agents expose conforming AgentCard and tasks/send endpoints.
The host machine has Python 3.11+, pip, and network access available.
3. System Architecture
3.1 Architectural Overview
DevPilot follows a layered architecture with four internal layers and two external protocol boundaries. All layers communicate vertically; no layer skips a neighbour. External MCP servers and A2A peer agents are connected via open standard protocols and are treated as external services.
3.2 Layer 1 ΓÇö Interface Layer
Responsibilities: accept user input (keyboard or A2A HTTP), render agent output (text, tool traces, diffs, errors), and manage the session lifecycle (start, save, resume, exit).
Terminal REPL: built with prompt_toolkit for history, autocomplete, and multiline input.
Output Renderer: uses Rich library for panels, syntax-highlighted diffs, collapsible tool traces, and colour-coded status.
A2A Task Receiver: a FastAPI endpoint that accepts incoming A2A tasks and forwards them into the same agentic loop used for interactive sessions.
3.3 Layer 2 ΓÇö Orchestration Layer
Responsibilities: run the agentic loop, maintain conversation history, manage context window size, and coordinate A2A client calls.
Agentic Loop: iterates model-call ΓåÆ parse response ΓåÆ execute tools ΓåÆ append results ΓåÆ repeat, until no tool_use blocks remain or max_iterations is reached.
Conversation History: stores the full message thread as a Python list; serialised to JSON for session persistence.
Context Manager: monitors token count and truncates the oldest non-system messages when the context window limit is approached.
A2A Client: discovers peer agents via their AgentCard, delegates subtasks, consumes SSE streams, and injects resulting artifacts as tool_result messages.
3.4 Layer 3 ΓÇö MCP Tool Layer
Responsibilities: connect to configured MCP servers, discover available tools, maintain a unified tool registry, execute tool calls (local or proxied), and enforce the permission guard.
MCP Client: connects to each server in mcp_servers.json on startup and calls tools/list to retrieve schemas.
Tool Registry: a dict mapping tool name to (schema, executor) pairs; built-in tools and MCP tools are stored identically so the agentic loop needs no awareness of the distinction.
Tool Executor: for built-in tools, executes Python code directly; for MCP tools, proxies a tools/call request to the originating server.
Permission Guard: intercepts write_file and run_bash calls; prompts the user (or blocks with error if in --no-confirm mode without the flag).
3.5 Layer 4 ΓÇö Provider Layer
Responsibilities: format conversation history and tool schemas into the correct API request body, send requests to the model API, parse the response, and identify text versus tool_use blocks.
AnthropicProvider: uses the official anthropic Python SDK; maps the shared message format to Anthropic's messages.create() call.
OpenAIProvider: uses the official openai Python SDK; maps the same shared format to chat.completions.create() with tools=[...].
Response Parser: normalises Anthropic and OpenAI response structures into a common list of (type, content) blocks consumed by the agentic loop.
3.6 External Integrations
MCP Servers
DevPilot connects to MCP servers over stdio or HTTP SSE transport. Supported servers include the official filesystem MCP server, GitHub MCP server, any third-party MCP server, and custom user-built servers. The client is transport-agnostic via the MCP Python SDK.
A2A Agent Network
DevPilot can communicate with any A2A-compliant agent. As a client it fetches the peer's AgentCard, sends a task, and consumes the SSE stream. As a server it publishes its own AgentCard at /.well-known/agent.json and handles incoming task requests. The A2A server runs on a configurable port (default 8000) alongside the terminal REPL.

6. Agile Development Plan
6.1 Methodology
DevPilot is developed using Agile Scrum with 2-week sprints. Each sprint delivers a working vertical slice of functionality that can be demonstrated and evaluated independently. The backlog is maintained as a prioritised list of user stories estimated in story points (Fibonacci scale). Acceptance criteria define the definition of done for each story.
Sprint cadence: 2 weeks per sprint, 4 sprints total (8-week project).
Story point scale: Fibonacci ΓÇö 1, 2, 3, 5, 8, 13.
Team velocity target: 25ΓÇô35 points per sprint.
Ceremonies: Sprint Planning (Day 1), Daily Standup (15 min), Sprint Review (Day 14), Retrospective (Day 14).
Tools: GitHub Issues (backlog), GitHub Projects (Kanban board), GitHub Actions (CI).
6.2 Epics
Epic 1 ΓÇö Core CLI Agent: terminal REPL, agentic loop, local built-in tools, permission guard.
Epic 2 ΓÇö Multi-Provider & UX: Anthropic/OpenAI/Ollama switching, rich output, session persistence, CI mode.
Epic 3 ΓÇö MCP Integration: MCP client, tool discovery, execution proxy, graceful disconnection.
Epic 4 ΓÇö A2A Protocol: A2A server endpoint, A2A client delegation, SSE streaming.
